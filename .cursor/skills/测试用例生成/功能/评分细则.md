# 功能用例质量评分（F1～F8）

**时机**：仅当二次 Check **通过** 后执行；Check 不通过则简报标注「未评分」。

**总分 = F1 + … + F8 = 100**（各维独立计分）。

## 与用户考量点的对应

| 用户考量 | 评分维度 | 上限 |
|----------|----------|------|
| 格式 | **F1** 格式规范 | 10 |
| 数量 / 覆盖 | **F2** 数量与覆盖 | 15 |
| 正向 / 反向 | **F3** 正反向结构 | 15 |
| 步骤可执行 | **F4** 步骤可执行性 | 15 |
| 预期可验证 | **F5** 预期可验证性 | 15 |
| 数据脱敏 | **F6** 数据脱敏 | 10 |
| 业务流程 | **F7** 业务流程关联 | 10 |
| 需求追溯 | **F8** 需求可追溯 | 10 |

> **F8 说明**：评估 alignment-matrix、功能点 FP 列、scope 矩阵是否对齐——与 Check 门禁互补。

## 档位

| 档位 | 总分 | 含义 |
|------|------|------|
| 优秀 | 90～100 | 样板级 |
| 良好 | 85～89 | **推荐交付目标** |
| 合格 | 75～84 | 可交付，须列改进建议 |
| 偏低 | 60～74 | 建议 1 轮质量提升后复评 |
| 不合格 | <60 | 质量不达标 |

## 生成时高分导向（Agent 必遵）

| 维度 | 高分做法 | 低分陷阱 |
|------|----------|----------|
| F1 | 9 列核心齐全、文首需求来源、FUNC-*-NNN 规范 | 缺列、编号随意 |
| F2 | 多模块覆盖、P0 充足、scope FP 全落功能点列 | 模块单一、无 P0 |
| F3 | 正向+反向配对、关联用例互指、反向预期互异 | 反向复制粘贴同一预期 |
| F4 | 步骤 `1. 2.` 编号、面向用户操作 | 写 GET /api/、HTTP 200 |
| F5 | 预期可观察、断言来源填 PRD § | 大量「待确认」占位当必过 |
| F6 | 无真实 URL/口令/手机号 | 明文敏感信息 |
| F7 | 含端到端/冒烟串联、P0 跨核心模块 | 孤立单点、无链路 |
| F8 | alignment-matrix 与 cases 编号一致 | 无追溯、FP 列缺失 |

## 低分处理

| 总分 | 动作 |
|------|------|
| ≥85 | 可交付 |
| 75～84 | 交付 + 简报 §10 改进建议 |
| <85 | Agent **应执行 1 轮质量提升**（按扣分项修正）→ 复评 → 更新简报 |
| <75 | Check 虽过，**质量不达标**，须继续优化后再交付 |

## 执行

Check 通过后自动调用：

```bash
cd 6.测试用例生成
python scripts/func_case_quality_score.py --topic 课程中心
```

或由 `func_case_second_check.py` 在 Check PASS 时内嵌评分并写入简报 **§9**。

详细计分规则见脚本 `func_case_quality_score.py` 内各 `_score_f*` 方法。
