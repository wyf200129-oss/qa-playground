# 功能测试用例生成（子 Skill）

> **前置**：已由 [测试用例生成/SKILL.md](../SKILL.md) 路由至本 Skill；并已读 [通用原则.md](../通用原则.md)。  
> **落盘 Rule**：[`.cursor/rules/功能用例生成-落盘路径.mdc`](../../rules/功能用例生成-落盘路径.mdc)

## 材料依据

1. **必须先读取** PRD、需求文档或用户提供的验收说明。
2. 无 PRD 时：仅输出 **待确认-产品** 清单，**禁止** 编造业务规则或必过预期。
3. 可引用接口契约了解系统能力边界，但 **功能用例预期以 PRD 为准**，不写 HTTP 层断言。

## 与接口用例的差异

| 维度 | 功能用例（本 Skill） | 接口用例（api 子 Skill） |
|------|---------------------|-------------------------|
| 视角 | 用户/业务操作 | HTTP path + method |
| 材料 | PRD | OpenAPI |
| 步骤 | 页面或业务流程 | headers / body / response |
| 覆盖 | 场景与验收点 | 每个 operation × 4 条 |
| 质量门禁 | 二次 Check + F1～F8 评分 | 二次 Check + S1～S8 评分 |

## 输出模板（四段式 + 扩展列）

| 列 | 说明 |
|----|------|
| 用例编号 | 如 `FUNC-{模块}-{序号}` |
| 模块 | 功能模块名 |
| 用例标题 | 一句话场景 |
| 前置条件 | 环境/账号/数据状态 |
| 测试步骤 | 编号步骤，面向用户操作 |
| 预期结果 | 可观察的业务结果 |
| 优先级 | P0 / P1 / P2 |
| 断言来源 | PRD / 产品说明 |
| 待确认 | `待确认-产品：…` 或 `待确认-开发：…` |

**扩展列（推荐，大项目）**：`用例类型`（正向/反向）、`关联用例`、`功能点`（FP-xxx，对齐 scope 矩阵）

## 编号规范

```text
FUNC-{模块缩写}-{三位序号}
反向配对：FUNC-{模块缩写}-{三位序号}-R
示例：FUNC-ORDER-001 / FUNC-NAV-001-R
```

## 落盘

> 路径相对于**工作区根目录**。

```text
功能用例集/{主题}/
  cases.md              # 用例主表（Markdown 表格）
  alignment-matrix.md   # 可选：需求 ID ↔ 用例编号
  scope-coverage-matrix.md  # 可选：功能点 FP 覆盖矩阵
  second_check_brief_功能用例_{YYYYMMDD_HHMM}.md  # 二次 Check 简报（脚本生成）
```

## 工作流

```
- [ ] 读取 PRD / 需求（或确认缺失）
- [ ] 列出 In Scope 功能点
- [ ] 每功能点至少 1 条正向；关键路径补充边界/异常
- [ ] 模糊处标 待确认-产品 / 待确认-开发
- [ ] 写入 功能用例集/ 
- [ ] **自动执行二次 Check**（见下文，最多 2 轮，可自动修复）
- [ ] **Check 通过后自动质量评分**（F1～F8，满分 100）
- [ ] 回复用例路径、简报路径、Check 结论、**质量总分/档位**；若 <85 按 §10 做 1 轮质量提升后复评
```

## 二次 Check（生成后必做）

用例落盘后 **必须** 运行二次 Check，不得跳过。

```bash
cd 6.测试用例生成
python scripts/func_case_second_check.py --topic {主题} --max-rounds 2
```

| 轮次 | 行为 |
|------|------|
| 第 1 轮 | 按 [检查清单.md](检查清单.md) 校验；发现问题 **自动修复** 后写回用例 |
| 第 2 轮 | 再次校验；仍不通过 → **FAIL**，生成失败简报，**禁止** 定稿交付 |
| 通过 | 生成成功简报 + 质量评分 → 可进入评审 / 接口用例 |

- 清单定义：[检查清单.md](检查清单.md)（F1～F12）
- 简报模板：[简报模板.md](简报模板.md)
- 简报目录：`功能用例集/{主题}/second_check_brief_功能用例_{YYYYMMDD_HHMM}.md`

**Agent 必须**：执行 Check 脚本 → 将简报路径、Check 结论、**质量评分** 回复用户；Check 不通过时说明 §6 遗留问题且 **不评分**。

## 质量评分（Check 通过后必做）

> **Check 管进门，评分管好坏**；Check 不过不评分。

| | 二次 Check | 质量评分 |
|--|------------|----------|
| 结论 | 通过 / 不通过 | **0～100 分** + 档位 |
| 时机 | 用例落盘后 | **仅 Check 通过** 后 |

### 八维基础分（F1～F8 = 100）

| 代号 | 维度 | 上限 | 你的考量对应 |
|------|------|------|--------------|
| F1 | 格式规范 | 10 | 格式 |
| F2 | 数量与覆盖 | 15 | 数量 / scope 覆盖 |
| F3 | 正反向结构 | 15 | 正向反向 |
| F4 | 步骤可执行性 | 15 | 步骤面向用户 |
| F5 | 预期可验证性 | 15 | 预期可观察 / PRD 追溯 |
| F6 | 数据脱敏 | 10 | 数据脱敏 |
| F7 | 业务流程关联 | 10 | 端到端 / 冒烟 |
| F8 | 需求可追溯 | 10 | alignment / FP 列 |

细则与 **高分导向**：[评分细则.md](评分细则.md)

### 档位与 Agent 行为

| 总分 | 档位 | Agent 行为 |
|------|------|------------|
| ≥85 | 良好+ | 可交付；生成时以此时为默认目标 |
| 75～84 | 合格 | 可交付，简报 §10 列改进项 |
| <75 | 偏低/不合格 | **须 1 轮质量提升** 后复评 |

**生成时主动靠拢高分**：步骤编号、预期可观察、少待确认、反向互异、FP 列对齐 scope。避免 HTTP 细节、P0 占位预期、反向复制粘贴。

```bash
# Check 已内嵌评分；也可单独跑：
python scripts/func_case_quality_score.py --topic {主题}
```

## 自检（生成时，Check 前）

- [ ] 步骤可被人手工执行，无 HTTP 细节堆砌
- [ ] 预期均可在 PRD 中找到依据，或已标待确认
- [ ] 无真实敏感信息
- [ ] 未混入接口用例表头（path/method/headers）
