# 接口用例质量评分（S1～S8）

**时机**：仅当二次 Check **通过** 后执行；Check 不通过则简报标注「未评分」。

**总分 = S1 + … + S8 = 100**（各维独立计分）。

## 与用户考量点的对应

| 用户考量 | 评分维度 | 上限 |
|----------|----------|------|
| 格式 | **S1** 格式规范 | 10 |
| 数量 | **S2** 数量与覆盖 | 15 |
| 正向 / 反向 | **S3** 正反向结构 | 15 |
| 数据精准性 | **S4** 数据精准性 | 15 |
| 响应断言 | **S5** 响应断言质量 | 15 |
| 数据脱敏 | **S6** 数据脱敏 | 10 |
| 业务流程 | **S7** 业务流程关联 | 10 |
| （建议补充）可执行 / 可追溯 | **S8** 可执行性与可追溯 | 10 |

> **S8 说明**：评估用例能否直接转 Requests、数据文件是否可复现、body 引用是否完整——这是「能跑起来」与「能审计」的维度，与 Check 门禁互补。

## 档位

| 档位 | 总分 | 含义 |
|------|------|------|
| 优秀 | 90～100 | 样板级 |
| 良好 | 85～89 | **推荐交付目标** |
| 合格 | 75～84 | 可交付，须列改进建议 |
| 偏低 | 60～74 | 建议 1 轮质量提升后复评 |
| 不合格 | <60 | 质量不达标 |

## 生成时高分导向（Agent 必遵）

| 维度 | 高分做法 | 低分陷阱 |
|------|----------|----------|
| S1 | 8 列表头、文首 version、编号 `_001~_004` 规范 | 缺 version、path 带基址 |
| S2 | 契约每个 operation ×4 全覆盖 | 漏接口、条数不足 |
| S3 | 反向 **触发条件互异**；预期文案不复制粘贴 | 3 条反向同一预期描述 |
| S4 | yaml 含 required 字段；neg 与 pos 数据不同 | 缺字段、neg 复用 pos 数据 |
| S5 | HTTP 码 + body 信息/字段要点（body 优先、联合 status）；反向含可绿设计；少待确认 | 仅写状态码、反向无 body 要点、大量待确认 |
| S6 | 统一 `[ENV_*]` / `[DEMO_*]` | 明文 password、真实 URL |
| S7 | login→Token→下单→支付 链路 _001 可串 | 孤立接口、无 Token 占位 |
| S8 | body `@数据文件`；yaml 含 method/notes | 内联大段 JSON、无 notes |

## 低分处理

| 总分 | 动作 |
|------|------|
| ≥85 | 可交付 |
| 75～84 | 交付 + 简报 §10 改进建议 |
| <85 | Agent **应执行 1 轮质量提升**（按扣分维修正）→ 复评 → 更新简报 |
| <75 | Check 虽过，**质量不达标**，须继续优化后再交付 |

## 执行

Check 通过后自动调用：

```bash
python scripts/api_case_quality_score.py --domain blackcore-mall
```

或由 `api_case_second_check.py` 在 Check PASS 时内嵌评分并写入简报 **§9**。

详细计分规则见脚本 `api_case_quality_score.py` 内 `DIMENSION_RULES` 注释。
