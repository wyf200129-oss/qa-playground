# pytest 失败修复闭环

> **统领关系**：本 Skill 为 **pytest 执行 → 失败自动诊断修复** 的总流程。  
> **用例修复四种方式总纲**（①红→绿 ②维持红 ③xfail ④skip）→ [接口自动化实现/用例修复四种方式.md](../接口自动化实现/用例修复四种方式.md)  
> **技术细节**（Traceback 阅读、fixture 链、栈范围）→ [代码调试与修改](../代码调试与修改/SKILL.md)  
> **fixture / 用例结构**（层数、scope、ids）→ [pytest-fixture与用例实现](../pytest-fixture与用例实现/SKILL.md)  
> **契约路径** → `6.测试用例生成/docs/contracts/`（见 [接口用例生成 Rule](../../rules/接口用例生成-契约优先.mdc)）

## 何时启用

- 用户要求 **运行 pytest / 测试用例 / 回归**
- 用户提及 **失败修复闭环、红转绿、三轮修复**
- 测试执行 **出现任何失败**（自动进入本流程，无需二次询问是否修复）

## 用例修复四种方式（与本 Skill 的关系）

pytest 执行后，**每条用例仅且必须** 归入以下四种之一：

| # | 方式 | 本 Skill 中的处理 |
|---|------|-------------------|
| **① 红→绿** | 根据代码逻辑优化测试，修复失败 | **主路径**：≤3 轮最小改动 + 回归 |
| **② 维持红** | 不优化或优化失败 | 需求不明确即停；3 轮仍失败 → 升级人工，保持 FAILED |
| **③ xfail** | 执行后优化：功能已有，实返与契约/PRD 不符 | 不编造改断言 → 加 mark，repair §10 |
| **④ skip** | 执行后优化：功能未完整实现 | 不编造 → 加 mark，repair §10 |

总纲 → [用例修复四种方式.md](../接口自动化实现/用例修复四种方式.md)；③④ → [xfail与skip治理.md](../接口自动化实现/xfail与skip治理.md)。

## 硬约束

### 1. 最大三轮修复

- 每条失败用例最多 **3 轮**「诊断 → 最小改动 → 回归」。
- 第 3 轮仍失败 → **停止自动改代码**，升级人工（见 [升级人工模板](#升级人工模板)）。
- 每轮须记录：现象、根因、改动文件/行、回归结果。

### 2. 最小范围改动

- **单文件可修时，不得改动其他文件**；避免维护成本增加。
- 尽量 **最少行数** 修复；保持整体结构与代码处于 **人工可控** 状态。
- 需动第 2 个文件时，须在本轮诊断中写明 **「单文件无法修复的原因」**（与 Rule [代码修改-先方案后执行](../../rules/代码修改-先方案后执行.mdc) §6 一致）。
- **禁止**：重构、换框架、顺手优化、批量改关联文件、无 trace 的玄学修改。

### 3. 代码改动规则（禁止「为绿而绿」）

#### a. 结合 Traceback

- 修改须与 **Traceback 指向的文件、行号、异常类型与信息** 直接相关。
- 无 Traceback / 日志时：**不得** 猜测性改代码；先运行 `pytest path -v --tb=long` 复现。

#### b. 修复优先级（从高到低）

| 优先级 | 依据 | 典型路径 |
|--------|------|----------|
| 1 | **契约 / 定稿文档** | PRD、`6.测试用例生成/docs/contracts/`、功能/接口用例表、Swagger/OpenAPI |
| 2 | **被测业务执行逻辑** | 业务代码、接口实现、状态机语义 |
| 3 | **Traceback 报错事实** | 异常类型、栈帧、fixture 链 |
| 4 | **被执行的测试代码** | 断言、测试数据、fixture 调用 |

- 仅当上层依据支持「测试写错」时，才改测试代码。
- 契约/PRD **未覆盖** 的点标 **待确认**，不得编造规则。

#### c. 业务缺陷 → 记录失败，不强修

- 若失败根因为 **产品/业务实现问题**（非测试误写、非环境/数据偶发），**不得** 为通过而改业务或改断言。
- 标注该用例为 **❌ 失败（产品缺陷 / 待开发确认）**，或经对比契约/PRD 后标 **`@pytest.mark.xfail`**（功能已有、行为不符）。
- 功能 **未实现**（如错误路径仅 500）→ **`@pytest.mark.skip`**，禁止编造 400 断言。
- reason **须含**：材料规定、实际行为、差异、**证据**、对接负责人。见 [xfail与skip治理.md](../接口自动化实现/xfail与skip治理.md)。

#### d. 不编造

- 不得虚构 PRD、契约字段、状态码、业务规则。
- 读不到契约/PRD 时，只列 **待确认项** 与验证步骤，不得当作确定结论改代码。

### 4. 需求不明确 → 立即终止 loop

若 **无法界定需求的实际内容**（需求不明确、内容无法定义），**不得** 进入或继续修复轮次：

- **立即结束** 该用例的修复 loop（不消耗三轮额度做猜测性尝试）。
- **返回失败**，标注 **❌ 失败（需求不明确）**。
- **必须写明具体失败原因**，至少包含：
  - 哪条预期/断言无法从现有材料推导；
  - 已查找但未找到的依据（契约 / PRD / 用例表路径，或「仓库内无对应文档」）；
  - 为何无法在不编造的前提下确定正确行为；
  - 建议人工补充的材料或确认项。

**典型触发**（任一即终止）：

| 情形 | 示例 |
|------|------|
| 契约/PRD 缺失且 traceback 不足以定预期 | 应用例表断言 201，仓库无该接口契约 |
| 多源冲突且无法判定以何为准 | PRD 写 200、契约写 201，无定稿说明 |
| 关键字段/状态码/业务语义未定义 | 失败涉及 enum 取值，文档未声明合法值 |
| 修复路径依赖未给出的业务决策 | 「非法输入应 400 还是 422」文档均未约定 |

**与 §3-c 区别**：§3-c 为 **需求可界定、实现不符合**；本节为 **需求本身不可界定**，禁止猜改。

**处置**（须 **pytest 已执行** 且有 trace/响应事实）：无法红→绿且不能编造 → 在 **② 失败修复闭环** 或后续 **① 二次 check** 中加 xfail/skip，并写入 repair 简报 **§10** / second_check **§8**。写用例阶段仅 **待确认**。见 [xfail与skip治理.md](../接口自动化实现/xfail与skip治理.md)。

## 总工作流

```
执行 pytest（目标范围）
    ↓
有失败？
    ↓ 否 → 结束（报告：全绿）
    ↓ 是
对每条 FAILED 用例：
    ↓
第 N 轮（N = 1..3）
    1. 读 Traceback + fixture 链（见 代码调试与修改 Skill）
    2. 查契约/PRD/用例表（若有）
    3. 判定根因类别：测试误写 | 数据/环境 | 业务缺陷 | 需求不明确 | 不确定
    4. 若需求不明确 → 立即终止 loop，返回失败（见 §4），下一条
    5. 若业务缺陷 → 不修复，标注失败；**执行后优化** → **xfail**（见 xfail与skip治理 §4.2），下一条
    6. 否则：单文件最小改动 → 回归该用例及合理范围
    ↓
仍失败且 N < 3 → N+1 轮
仍失败且 N = 3 → 升级人工；或经对比实返与契约 → **xfail/skip**（禁止编造改断言）
    ↓
汇总：通过 / failed / xfailed / skipped / 人工升级
    ↓
【执行后优化】repair 简报 §10 登记 xfail/skip；接口自动化另与 second_check §8 对齐
```

## 每轮输出模板

```text
【第 N 轮 / 用例：test_xxx】
1. 现象：{一句话}
2. Traceback：
   - 异常：{Type}: {message}
   - 位置：{file}:{line}（fixture 链：…，若适用）
3. 根因类别：{测试误写 / 数据环境 / 业务缺陷 / 需求不明确 / 不确定}
4. 依据：{契约 §… / PRD … / traceback / 业务代码 …，或「需求无法界定」说明}
5. 修改：{文件:行，或「不修改—业务缺陷」，或「不修改—需求不明确」}
6. 回归：{命令} → {结果}（需求不明确时可省略回归）
```

## 需求不明确 — 立即失败模板

**不进入修复轮次**，直接输出：

```text
【❌ 失败 — 需求不明确】用例：test_xxx
1. 失败原因：{一句话说明无法界定的内容}
2. 无法推导的预期：{具体断言/字段/状态码/业务规则}
3. 已查依据：
   - {契约/PRD/用例表路径} → {未找到 / 未覆盖 / 与实现冲突}
4. 为何不能修复：{不编造前提下无法确定正确行为}
5. 待人工补充：
   - {需确认的文档、决策人或业务规则}
```

## 升级人工模板

第 3 轮仍失败，或需人工决策时：

```text
【❌ 失败 — 待人工】用例：test_xxx
1. 初始现象：（trace 摘要，脱敏）
2. 已尝试（3 轮摘要）：轮次 / 改动 / 结果
3. 残余根因假设：（诚实标注不确定）
4. 人工排查思路：
   - 建议检查 …
   - 建议对照契约 …
5. 残余风险：…
```

课堂/作业可对照：`第九节课/第09节-pytest失败修复闭环与一页小结.md`

## 执行后简报（必产出）

跑测 + 修复（或终止）结束后，**必须**按 [简报模板.md](简报模板.md) 生成 Markdown 并落盘至 `第九节课/报告集/`（或用户指定目录，如 `{框架}/reports/`）。

**Allure（框架已接入时）**：最终回归通过后须重新 `--alluredir` + `allure generate`，并在简报或回复中给出 **Allure HTML 路径**（见 [Allure报告规范.md](../接口自动化实现/Allure报告规范.md)）。

```text
Task Progress:
- [ ] 执行 pytest，记录命令与时间（精确到秒）
- [ ] 修复闭环（≤3 轮 / 需求不明确即停）
- [ ] 最终回归
- [ ] 生成简报 MD（§1～§10，用例级 a～f；§10 执行后 xfail/skip）
- [ ] **（若已接 Allure）** regenerate allure-report 并记录路径
- [ ] 回复用户简报路径、Allure 路径（若有）与总体结论
```

## 与 Rule「先方案后执行」的关系

- **本 Skill 激活且用户已要求执行测试时**：每轮修复前输出 **当轮诊断摘要**（用上表模板），**轮次间无需等待用户逐轮确认**；授权范围限于 **失败用例的最小修复**，不得扩 scope。
- **用户仅咨询、未要求跑测/修复时**：仍遵守 Rule [代码修改-先方案后执行](../../rules/代码修改-先方案后执行.mdc)（先方案 → 明确同意 → 再改文件）。

## 自检清单

- [ ] 需求不明确时已立即终止 loop，未做猜测性改代码
- [ ] 已运行 pytest 并基于真实 Traceback 诊断
- [ ] 修复轮次 ≤ 3；超限已升级人工并标注 ❌ 失败
- [ ] 单文件可修时未动其他文件
- [ ] 已按优先级 b 选择修改对象；未「为绿而绿」
- [ ] 业务缺陷未强行改代码
- [ ] 未编造契约/PRD 未声明的规则
- [ ] 已给出回归命令与结果
- [ ] 已生成并落盘执行修复简报（见 [简报模板.md](简报模板.md)）
- [ ] **（若框架已接 Allure）** 已 regenerate 并回复 allure-report 路径
- [ ] xfail/skip 已附 reason + 证据（见 xfail与skip治理.md）

## 反例（禁止）

- 需求缺失时凭「惯例 REST」猜状态码并改断言
- 断言失败却将 mock 返回值从 400 改成 200，只为让负向用例变绿
- 无 Traceback 就重构 `conftest.py`
- 第 1 轮失败后未记录原因，连续大范围重写测试文件
- 3 轮失败后仍 silent 继续改代码
- 写用例/骨架阶段预设 `@pytest.mark.xfail` / `skip`（须等 pytest 执行后在二次 check / 修复闭环 / 简报中决策）
- 引用不存在的 PRD 条款或契约字段作为修改依据
