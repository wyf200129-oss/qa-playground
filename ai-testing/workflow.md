## 🔄 工作流程

```
业务需求文档 → AI 解析需求要点 → AI 辅助生成测试场景
                                    ↓
                              人工评审 + 补充边界
                                    ↓
                              AI 根据场景生成用例初稿
                                    ↓
                              人工审查 + 补充异常路径
                                    ↓
                              纳入用例库 → 代码化 → Jenkins 回归
```

---

## 🧠 核心思路

### Step 1：需求解析

**输入**：一段来自产品经理的需求描述（已脱敏）

```
需求示例：新增"销售订单审核"功能，订单提交后需经过主管审批，
审批通过后进入发货流程，驳回则退回草稿状态。需支持多级审批链，
审批人可加签、转审。
```

**思路**：将需求丢给 AI 前，先写好一段 System Prompt 限定角色和输出格式。

```markdown
你是一个资深测试工程师，请对以下需求进行分析：

1. 提取核心功能点
2. 识别可能的业务异常场景
3. 列出需要关注的边界条件
4. 按功能模块 + 场景分类输出

输出格式为 Markdown 表格。
```

### Step 2：场景扩展

AI 会输出类似这样的结构化场景列表：

| 模块 | 场景 | 优先级 |
|------|------|--------|
| 销售订单审核 | 单级审批-审批通过 | P0 |
| 销售订单审核 | 单级审批-审批驳回 | P0 |
| 销售订单审核 | 多级审批-逐级通过 | P1 |
| 销售订单审核 | 审批加签 | P2 |
| 销售订单审核 | 转审 | P2 |
| 异常 | 并发审批冲突 | P1 |
| 边界 | 审批链最长层级 | P1 |
| 边界 | 空审批意见提交 | P1 |

### Step 3：用例生成

对每个场景，让 AI 生成：

- 前置条件
- 测试步骤
- 预期结果
- 测试数据

然后人工过一遍——补充等价类、边界值、异常流程。

### Step 4：入库 + 自动化

- 确认后的用例纳入用例库
- 高频场景（订单提交、审批通过、驳回）优先代码化
- 接入 Jenkins 定时回归

---

## 🔑 关键经验

### ✅ 做对了什么

1. **先定规则，再丢需求**：给 AI 写好 System Prompt（角色 + 输出格式 + 思维链），比直接问效果好很多
2. **用 RAG 喂业务上下文**：把已有的需求文档、数据库表结构、接口文档作为上下文喂给 AI，生成的用例更贴近实际
3. **AI 做 80%，人做 20%**：大量场景让 AI 覆盖，人工专注于边界值和异常流程
4. **结构化输出**：要求 AI 输出固定格式，方便后续转成测试管理平台可导入的格式

### ⚠️ 注意什么

1. **不能完全依赖**：AI 生成的用例需要人工审查，特别是权限校验、数据一致性等强逻辑场景
2. **脱敏**：需求文档和数据库表结构喂给 AI 前要脱敏
3. **迭代**：Prompt 需要根据输出质量持续调优，不是一次写完就完事

## 🛠️ 工具选型对比

| 工具 | 优势 | 适用场景 |
|------|------|----------|
| ChatGPT | 推理能力强，适合复杂逻辑分析 | 需求解析、边界识别 |
| Kimi | 上下文窗口大，适合大段文档 | 长需求文档批量处理 |
| ChatGPT + Skills/Rules | 可固化测试规则 | 标准化输出 |

---

> 💡 **结论**：AI 不是替代测试工程师，而是把我们从重复性工作中解放出来，把精力集中在业务理解和边界设计上。
